# Sprint Planning Evaluation Strategy

## Goal

Evaluate whether `sprint-planning` turns a sized and risk-aware task set into a realistic iteration commitment instead of a wish-list backlog dump.

## Why This Skill Matters

`task-breakdown` creates the work items. `estimation` and `risk-assessment` expose size and uncertainty. `sprint-planning` should make a defensible short-horizon commitment that respects both capacity and risk.

The key question is whether the skill:

- starts from real capacity instead of desire
- chooses the smallest scope that fits the sprint goal
- keeps dependency order and major risk visible
- distinguishes committed scope from stretch work and deferred work

## Initial Evaluation Mode

The first evaluation is a **manual routed handoff review** using a bounded brownfield planning slice.

This is review-stage evidence only. It does not replace future isolated automated benchmarks or live sprint execution evidence.

## Scenario

- `access-review-modernization-sprint-plan`

Inputs:

- task list
- estimate set
- risk register
- capacity note

## Assertions

1. **capacity-first**
   - the output starts from actual capacity and carry-over constraints

2. **scope-is-defensible**
   - committed work fits the sprint goal without stuffing too many risky items into one iteration

3. **sequence-respects-dependencies**
   - work ordering follows task and risk dependencies instead of flat priority ranking

4. **stretch-and-defer-are-explicit**
   - committed, stretch, and deferred work are separated clearly

5. **does-not-collapse-into-backlog-prioritization**
   - the output remains an iteration plan rather than a generic roadmap or priority list

## Pass Standard

Treat the skill as strong review-stage evidence if the handoff artifact shows a clear sprint-commitment boundary downstream of estimation and risk assessment.
